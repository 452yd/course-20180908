## Первое два задания: преобразования XML <-> DBD

### Исходные материалы:

1. XML-представления БД [TASKS](./materials/tasks.xdb) и [PRJAdm](./materials/prjadm.xdb)
2. [DDL](./materials/dbd_const.py ) для генерации DBD

### Требования

1. Классы *RAM-представления*, описывающие элементы схемы данных, разместить в отельном модуле.
1. Функции преобразования:

    * XDB -> RAM-представление
    * RAM-представление - XDB
    * RAM-представление -> DBD
    * DBD -> RAM-представление

    должны быть:

    1. Размещены в отдельном пакете
    2. Разнесены по модулям (открытый вопрос, обсуждение)

1. По второй задаче:

    1. XML-атрибуты в результатах работы программы по второй задаче должны идти в заданном порядке.
    2. XML-тэги таблиц должны быть отсортированы по латинскому имени, полей - по русскому.

1. **[M1]** Тестовое приложение, автоматизированно выполняющее преобразование *эталонного XDB* -> DBD -> XDB1 и контролирующее идентичность XDB1 *эталонному XDB*.

### Требования к коду

1. [PEP08](https://www.python.org/dev/peps/pep-0008/).
   Статус: конвенция, стандарт. Отклонения не допускаются.
2. Функциональность приложения разместить в отдельном классе.
3. Для исключений приложения определить специальный класс.
4. Для обработки параметров командной строки использовать модуль `argparse`.

### Задачи

#### Примерный план мероприятий

* [#1] Описание алгоритмов решения задач 1 и 2 - 2 ч.
* [#2] [Знакомство с инфраструктурой](https://docs.python.org/3/tutorial/index.html) - 8 ч.

    Вопросы к рассмотрению:

    * Общая структура программы
    * Определения функций, классов
    * Определение и обработка исключений
    * Структуры данных: списки (массивы), словари (ассоциативные массивы)

* [#3] (#1-2) Преобразование XDB -> RAM-представление - 8 ч.
* [#4] (#3) Преобразование RAM-представления -> XDB - 4 ч.
* [#5] (#3) Преобразование RAM-представление -> DBD - 8 ч.
* [#6] (#5) Преобразование DBD -> RAM-представление - 4 ч.
* [#7] (#3-6) Создание тестового приложения *[M1]* - 2 ч.

#### Рекомендации

Задачу "Знакомстиво с инфраструктурой" лучше сразу не закрывать, а вносить в неё затраты, связанные с темой задачи, возникающие в контексте других задач.

Например: Вы ознакомились с *git*, прочитали вводный курс по Python, установили PyCharm, указали затраты в задаче `#2` в таком виде:

> Установка и изучение материалов по Git - 3 ч.
> Изучение материалов по Python - 2 ч.
> Установка PyCharm и первоначальное знакомство с этой IDE - 1 ч.

Далее, Вы приступаете к решению первой задачи, и выясняете, что предварительно Вам следует понять, как обрабатывать XML.
Вы делаете соответствующие запросы в Google, обнаруживаете, что поддержка XML есть в Python "из коробки", и читаете, как применять [minidom](https://docs.python.org/3/library/xml.dom.minidom.html#module-xml.dom.minidom). Допустим, на всё это у Вас ушло 2 часа.
Вы снова открываете задачу `#2`, и указываете затраты:

> Изучение API модуля `minidom` - 2 ч.

Продалжая работы по первой задаче, Вы обнаруживаете, что Вам нужно ещё научиться работать с SQLite. Посредством Google Вы выясняете, что работа со всему СУБД в Python устроена однообразно, следуя спецификациям [PEP249](https://www.python.org/dev/peps/pep-0249/), а способы применения конкретно SQLite [подробно описаны](https://docs.python.org/3/library/sqlite3.html#module-sqlite3) в замечательной документации Python.
В итоге, на всё это Вы тратите 1,5 часа, и отражаете это в затратах по задаче `#2`:

> Изучение API Python-биндинга SQLite - 1,5 ч.

Таким образом, в задачах #3-7 Вы отразите только время, непосредственно ушедшее на их реализацию.
